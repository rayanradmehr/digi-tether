# Wallet Module — Architecture Decision Records

---

## ADR-WM-001: Wallet Pool Exists to Guarantee Sub-Second Assignment

**Status**: Accepted

**Context**

Customer onboarding must be fast. If the backend generated a new wallet
on-demand at assignment time, it would need to:
1. Create a SignerJob.
2. Wait for the Offline Signer to poll (up to 60 seconds).
3. Wait for the signing result to return.
4. Store the wallet.
5. Return the address.

This makes wallet assignment synchronous on Signer latency — unacceptable
for a customer-facing onboarding flow.

**Decision**

Maintain a pool of pre-generated `AVAILABLE` wallets per family.
Assignment is an instant read-and-update operation against the pool.
The pool is replenished asynchronously by the Signer in the background.

**Consequences**

- Assignment latency: < 50ms (DB operation only).
- Pool size must be monitored and maintained above the configured threshold.
- `WalletPoolLow` events must trigger alerting in production.
- Pool exhaustion (zero available wallets) is a critical incident — deployment
  must ensure pool health before going live.

## ADR-WM-002: Wallets Are Pre-Generated by the Offline Signer

**Status**: Accepted

**Context**

Wallet generation requires generating a private/public key pair.
The backend is permanently prohibited from handling private keys
(Architecture Rule §3: Private Key Isolation).
Therefore, wallet generation can only occur on the Offline Signer.

**Decision**

The backend creates `CREATE_WALLET` SignerJobs when the pool falls below
the replenish threshold. The Offline Signer generates the key pair,
computes the address, and returns only the public key and address in
the result. The private key never leaves the Signer.

**Consequences**

- Pool replenishment is asynchronous; throughput is bounded by Signer capacity.
- `batchSize` and `maxConcurrentJobs` must be tuned for each deployment.
- The backend stores `publicKeyFingerprint` and `signerVersion` for audit,
  but never stores or processes the private key.

## ADR-WM-003: Wallet Ownership Is Immutable

**Status**: Accepted

**Context**

Once a customer is assigned a wallet address, that address is permanently
linked to their identity on the blockchain. Re-assigning a wallet would:
- Create an audit trail conflict.
- Risk sending funds to the wrong customer.
- Violate the integrity of on-chain transaction history.

**Decision**

The `status → ASSIGNED` transition is one-way and final.
`customer_id` and `assigned_at` are set once and never modified.
The repository must enforce this at the persistence layer:
`update()` must never overwrite `customer_id` or `assigned_at`.

**Consequences**

- Wallets can never be recycled between customers.
- Pool consumption is strictly monotonic — only new Signer-generated
  wallets enter the AVAILABLE pool.
- Compromised wallets must be archived and new wallets assigned to the
  affected customer through a manual recovery procedure.

## ADR-WM-004: `customerId` Is Opaque — No Identity-Provider Coupling

**Status**: Accepted

**Context**

The current exchange uses Iranian National ID as a customer identifier.
Hardcoding `nationalId` as a column name or type would:
- Couple the backend to a single identity provider.
- Prevent future support for foreign customers, corporate accounts,
  or alternative identity schemes.
- Violate the Single Responsibility Principle (the backend owns blockchain
  infrastructure; identity is an Exchange concern).

**Decision**

The Wallet Module stores only `customerId: string` — an opaque,
identity-provider-independent token. The Exchange maps its internal
identity to this token before calling the Backend API.
The Backend never interprets the contents of `customerId`.

**Consequences**

- The Backend is compatible with any future identity scheme.
- The Exchange owns the mapping between National ID and `customerId`.
- If the Exchange changes identity providers, the Backend requires zero changes.
- `customerId` must be treated as PII for logging and data protection purposes.

## ADR-WM-005: Wallets Never Store Balances

**Status**: Accepted

**Context**

A wallet balance is not a property of the wallet itself — it is a
property of the blockchain ledger at a given point in time.
Storing balances in the `wallets` table would:
- Create stale data that must be invalidated on every transaction.
- Require synchronisation logic in the Wallet Module.
- Couple the Wallet Module to blockchain state, violating its boundary.
- Risk balance inconsistencies between the database and the blockchain.

**Decision**

The `wallets` table has no balance column. No balance cache exists.
Balances are computed on-demand by the Blockchain Sync Module (Phase 5+)
through direct blockchain queries or a maintained ledger.

**Consequences**

- The Wallet Module remains blockchain-agnostic.
- Balance queries require a separate module call — this is intentional.
- Downstream modules (Withdrawal, Sweep) must query the Sync Module,
  not the Wallet Module, for available funds.

## ADR-WM-006: EVM Chains Share a Single Wallet Family

**Status**: Accepted

**Context**

All EVM-compatible chains (Ethereum, BSC, Polygon, Arbitrum, Optimism,
Base, Avalanche C-Chain, etc.) derive wallet addresses identically:
keccak256(public_key)[12:] as a 20-byte hex address. A secp256k1
key pair produces the same address on every EVM chain.

Creating separate wallet pools per EVM chain would:
- Multiply pool size requirements (7+ EVM chains × 500 = 3500+ wallets).
- Require separate CREATE_WALLET jobs for each chain.
- Provide no security benefit (same address, same private key).
- Fragment inventory unnecessarily.

**Decision**

All EVM-compatible chains are grouped under a single `WalletFamily.EVM`.
One pool, one address space. Network context is resolved at sweep/withdrawal
time by the calling module via the `NetworkModule`.

**Consequences**

- Pool size for EVM = 500 (not 3500).
- A new EVM chain requires only a new `driverKey → EVM` mapping in
  `WalletFamilyResolver`; no schema changes, no new pool config.
- Non-EVM chains (TRON, Bitcoin, Solana) have separate families because
  their address derivation schemes are incompatible.

## ADR-WM-007: Backend Never Creates or Owns Wallets

**Status**: Accepted

**Context**

Wallet creation requires generating a cryptographic key pair. The backend:
- Has no access to a Hardware Security Module (HSM).
- Must not expose the secret generation environment to network requests.
- Must not introduce cryptographic library risk into the production backend.
- Must not store private keys under any circumstances.

**Decision**

Wallet creation is exclusively the responsibility of the Offline Signer.
The backend only stores the resulting address and public key fingerprint.
The backend triggers creation by publishing a `CREATE_WALLET` SignerJob;
it never calls any key generation function directly.

**Consequences**

- The backend is free of any cryptographic key-generation library.
- The attack surface for key extraction from the backend is zero.
- In the event of a backend compromise, no private keys are exposed.
- Creation throughput is bounded by Signer availability; this is acceptable
  because the pool buffers demand.

## ADR-WM-008: Optimistic Locking on Wallet Assignment

**Status**: Accepted

**Context**

Two concurrent API calls could simultaneously select the same
`AVAILABLE` wallet and assign it to different customers (phantom read).
This would violate the one-wallet-per-customer-per-family invariant.

**Decision**

The `wallets` table uses a TypeORM `@VersionColumn` for optimistic locking.
Additionally, `WalletRepository.findFirstAvailable()` uses
`FOR UPDATE SKIP LOCKED` to prevent phantom reads at the database level.
If a version conflict occurs, the service retries once before throwing
`WalletPoolExhaustedError`.

**Consequences**

- Wallet assignment is safe under concurrent load.
- Retry adds at most one round-trip per assignment under contention.
- Under extreme concurrency, callers may receive `WalletPoolExhaustedError`
  even when wallets are available — the retry mitigates this in practice.

## ADR-WM-009: Wallet Audit Log Is Append-Only

**Status**: Accepted

**Context**

Wallet lifecycle transitions (assign, lock, compromise, archive) are
financially and legally significant events. An auditable, tamper-evident
record is required for:
- Regulatory compliance.
- Incident response.
- Customer dispute resolution.

**Decision**

Every status transition writes an append-only row to `wallet_audit_log`.
No audit log row is ever updated or deleted.
The table has no `deleted_at` column.
Audit log retention is ≥ 7 years (configurable per jurisdiction).

**Consequences**

- Audit log grows unbounded — archival and partitioning strategy required in Phase 5+.
- A corrupt or missing audit log row is a data integrity incident.
- The `WalletAuditLogRepository.append()` is the only permitted mutation method.
